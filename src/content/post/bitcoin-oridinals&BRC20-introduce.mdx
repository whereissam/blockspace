### Ordinal 協議

Ordinal 協議是由 Casey Rodarmor 於 2023 年初提出的。Rodarmor 是一位比特幣開發人員，他一直致力於在比特幣上構建 NFT 的技術。

![image](https://hackmd.io/_uploads/Syg4DH3np.png)
Rodarmor 本人的樣子

Ordinals 本質上是為 NFT 設計，為了解決比特幣上 NFT 存在的一些問題，例如缺乏標準、擴展性差、功能有限，難以滿足複雜的 NFT 應用需求。傳統的比特幣 NFT 方案往往需要使用側鏈或其他第三方解決方案，這會降低安全性和可靠性

Ordinal 協議則是使用 taproot 裡面的 P2TR 腳本，直接在比特幣主網上構建 NFT，這使得 NFT 更加安全、可靠和易於使用。

> P2TR 腳本全名是 Pay-To-Taproot，細節就不過多贅述，有興趣的可以參考[這篇](https://blog.safeheron.com/blog/v/jian-ti-zhong-wen-2/safeheron-dong-cha/an-quan-lu-wen-zhang/bitcoin-taproot-sheng-ji-hou-dui-jiao-yi-yin-si-xing-dai-lai-de-bian-hua)

想像一下，你有一個巨大的樂高積木盒子，所有的樂高積木看起來都一模一樣。序數就像一個系統，可以讓你拿單一樂高積木並用記號筆在上面寫字。這使得這些樂高積木變得特別和獨特，儘管它們一開始是完全相同的。

在比特幣中，樂高積木是稱為 Satoshis（以比特幣創造者的名字命名）的微小單位。通常，所有中本聰都受到相同的待遇。但使用 Ordinals，您可以在 Satoshi 上“書寫”，添加圖片、一些文字甚至影片遊戲角色等資訊。書寫在 Satoshi 上的動作就叫做銘刻，這創建了一種直接儲存在比特幣網路上的獨一無二的數位項目。

問題是：有些人認為在這些樂高積木（序號）上書寫會使盒子（比特幣網路）變得混亂，並使每個人的速度變慢。這有點像在圖書館裡的所有東西上寫東西，對某些東西來說可能很酷，但它讓每個人都更難找到他們需要的東西。 因此，序號是一種向比特幣添加獨特內容的新方式，但從長遠來看這是否是一個好主意存在一些爭議。

1. 網路堵塞：想像一下，除了信件之外，你的信箱裡還塞了一堆額外的東西。序號將圖片和影片等數據添加到比特幣交易中，這可以使交易規模變得更大。這可能會減慢交易處理速度並增加每個使用比特幣的人的費用。
2. 意外用途：比特幣是為安全、快速的價值轉移而設計的。序號為儲存在區塊鏈上的潛在不必要的內容打開了大門，引發了對其長期可用性的擔憂。
3. UTXO 問題：比特幣使用稱為 UTXO（未花費交易輸出）的系統，其中交易使用特定輸出作為輸入。如果 UTXO 中充滿了大銘文，Ordinals 可能會使 UTXO 無法用於常規交易。這可能會讓一些資金滯留在錢包裡。
   - 通常情況下，你可以花掉錢包裡的任何 Satoshis。
   - 對於 Ordinals，如果 Satoshis 附有很大的銘文，它可能會變得太大而無法在常規交易中使用。
   - 這個「卡住」的 Satoshis（UTXO）仍然有價值，因為它位於比特幣區塊鏈上，但你可能無法輕易花掉它。 值得注意的是，Ordinals 是一項新技術，這些問題的嚴重程度仍在爭論中。有些人認為網路可以適應，而有些人則擔心長期後果。

### BRC20

[BRC-20](https://www.brc-20.io/)
[BRC-20 是什么？聊聊 BTC 链上的最新故事 【Vic TALK 第 502 期】](https://youtu.be/3Zmrirgk0IA?si=ZXoXv36WrAvP97PZ)
[What are BRC-20 tokens on Bitcoin?](https://mythbtc.xyz/brc-20-token-standard-bitcoin/)

由匿名開發者 Domo 基於 Ordinals 協議的功能構建，核心概念是利用 Ordinal Inscriptions 中的 JSON 資料。透過 BRC20 標準，使用者能夠輕鬆實現 Token 合約的創建（Deploy）、Token 的鑄造（Mint）以及 Token 的轉移（Transfer）等關鍵功能，就很像是在以太坊上面寫智能合約在創建代幣的過程，智能合約也提供了差不多的功能

用簡單的來講，比特幣是一個大帳本，在這個大帳本裡面又做了一個小帳本就是 BRC20 在做的事情，BRC20 代幣不是實際代幣，而是記錄了特定文本的聰，需要單獨的索引器（第三方服務）來了解 BRC20 代幣的狀態或餘額

BRC20 文本長得像這樣

```
{
  "p": "brc-20",
  "op": "deploy",
  "tick": "ordi",
  "max": "21000000",
  "lim": "1000"
}
```

![image](https://hackmd.io/_uploads/By7XIPl6p.png)

#### BRC20 的局限性

目前，BRC20 代幣的創建和轉移由第三方服務商處理。這是由於技術上的限制，創建和轉移 BRC20 代幣需要使用 Taproot 和 Ordinals 協議。然而，這些協議對於非程式設計人員來說可能有些晦澀難懂。這些第三方服務商可被視為是 BRC20 代幣的銀行，他們負責紀錄 BRC20 代幣交易。在他們完成記帳後，交易才會被傳回比特幣的大型帳本。然而，這樣做存在著中心化風險。若第三方服務商受到黑客攻擊，則 BRC20 代幣可能會遺失或無法使用。

所有的數據都被記錄在比特幣上，但要確認誰向誰轉移，就需要 indexer 的協助。這類似於以太坊上的區塊鏈瀏覽器。目前，indexer 大多由第三方機構實現，每個實現方式都有所不同。因此，可能導致數據不一致的情況發生。這一點需要特別留意。

### ORC20

- [ORC-20 協議官推](https://twitter.com/orc20protocol)
- [ORC-20 協議文檔](https://orc20protocol.gitbook.io/)
- [ORC-20 索引器](http://20scan.com/indexer)
- [ORC-20 轉移網站工具](http://wallet.20scan.com)
- [ORC-20 交易市場](https://preview.xrcswap.org/marketplace?or)

ORC20 的創建是為了解決 BRC20 標準的一些限制，特別是在比特幣區塊鏈上

#### 為什麼 BRC20 有限制：

1. 雙重支出：BRC20 代幣容易受到雙重支出的影響，即相同的代幣可能會被花費兩次。
2. 資料格式有限：最初部署 BRC-20 代幣時，總供應量和每次鑄造的最大代幣數量是固定的，無法更改
3. 可擴展性問題：隨著 BRC20 代幣數量的增加，網路可能會變得擁塞。

#### ORC20 如何解決這些問題：

- UTXO 模型：ORC20 利用比特幣的未花費交易輸出（UTXO）模型，消除了雙重支出的可能性。
- 更廣泛的數據支援：與 BRC20 相比，ORC20 代幣可以儲存更廣泛的資料格式。 提高可擴展性：從設計上來說，ORC20 的目標是比 BRC20 更具可擴展性。

主要特徵：

- 向後相容性：ORC20 努力與現有的 BRC20 代幣相容，以便更輕鬆地遷移。
- 靈活性：它提供了對代幣屬性的更多控制，例如初始供應量、最大供應量，甚至以後更新這些屬性的能力（這是一些人爭論的焦點）。
- 可自訂：ORC20 允許為創作者定義客製化功能，例如費用或版稅。

#### ORC20 是 BRC20 的替代品嗎？

- BRC20：更簡單的設計，可能更適合可替代代幣和價值儲存。
- ORC20：更複雜，但為獨特的代幣應用程式提供了更大的靈活性和安全性。

### SRC-20

SRC-20 是專為比特幣區塊鏈設計的代幣標準，類似以太坊上的 ERC-20 和幣安智能鏈上的 BRC-20。它透過利用 Stamps 協議來促進比特幣代幣的創建和管理。

#### SRC-20 提供的功能：

- 證券型代幣焦點：SRC-20 特別適合創建證券型代幣，代表股票或債券等現實世界資產的所有權。
- Stamps 協定整合：它建立在 Stamps 協定之上，允許在比特幣交易中嵌入資料。這使得可以儲存有關代幣的信息，例如其名稱、總供應量和所有權詳細資訊。
- 效率：SRC-20 交易旨在提高效率，利用壓縮技術最大限度地減少區塊鏈儲存需求。 S

#### RC-20 解決哪些問題？

- 比特幣的功能有限：傳統上，由於資料儲存的限制，比特幣並不是創建和管理代幣的理想選擇。 SRC-20 透過使用 Stamps 協定進行資料儲存來解決這個問題。
- 缺乏安全代幣標準：在 SRC-20 之前，比特幣上的安全代幣沒有廣泛採用的標準。 SRC-20 提供了一個用於建立和管理安全性令牌的框架，具有所有權追蹤等功能。

本質上，SRC-20 旨在釋放比特幣代表現實世界資產和創建更複雜金融工具的潛力。 以下是一些需要考慮的其他要點： 與 BRC-20 的比較：SRC-20 和 BRC-20 都可以在比特幣上創建代幣。然而，SRC-20 可能在資料壓縮和專注於安全令牌方面提供一些優勢。 發展階段：SRC-20是一個相對較新的標準，其在比特幣生態系統中的長期採用仍有待觀察。
